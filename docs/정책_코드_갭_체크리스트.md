## E. 골든타임 (~8분) 달성 가능성

| # | 정책 목표 | 병목 (코드) | 상태 |
|---|-----------|-------------|------|
| E1 | 열사병: 5+2+AI ≈ 8분 | 1단계 5분 + 관찰 2분은 **서버 상태머신** ✅ | ⚠️ |
| E2 | 앱이 5분 구간을 **놓치지 않음** | sync **10분**이면 1단계 타이머가 **한 번에 점프**할 수 있음 | ❌ 🔧 |
| E3 | 실신: ~3~4분 | −30% 즉시 관찰 + 2분 — sync 10분이면 **지연** | ❌ 🔧 |
| E4 | AI 무응답 2회 ~1분 | 30s + 30s timeout | ✅ |

**핵심 갭:** 정책의 5분·2분 타이머는 서버에 있으나, **입력 데이터가 10분마다만 들어오면 타이머가 정확히 동작하지 않음**.

## G. 우선순위 작업 제안 (P0 → P2)

### P0 — 감지가 아예 안 되는 경우

1. **C3** `SYNC_INTERVAL_MS` 10분 → **1~2분** (Android + RN + Worker)
2. **C4/C8** 운영: Samsung Health → HC 심박 공유 (앱만으로 해결 불가)
3. **C5** 위치 권한·GPS 확보 (전송 전제)

### P1 — 정책 정합·신뢰도

4. **A1** 온보딩 HC 7일 이력 → `heart_rate_history` 전송
5. **A3/A4** 7일 rolling baseline 갱신 배치 + `baseline_sigma` 저장
6. **E2** 서버: 마지막 `health_data` 시각 기준으로 phase `enteredAt` 보정 (또는 sync 주기 단축)
7. **D2** Twilio ring timeout 60초 vs 30초 기획 확정

### P2 — 운영·UX

8. **B10** 상태머신 영속화
9. **A5** 설정 화면 실제 baseline/갱신일
10. **F3** 「기기 연결」→ Galaxy Wearable/블루투스 설정 직링크
11. **B10** 서버 재시작 시 observing/alert 복구
